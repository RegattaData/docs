## Configuration Planning

The first step is defining the system's topology, which is determined by your workload profile, performance goals, and storage requirements. Since RegattaDB is highly versatile, we can tailor this configuration to meet specific needs.

Below are several common configurations.

#### Single Node

In a single node deployment, there is only one option:

<Frame>
  ![Image](/images/image-1.png)
</Frame>

#### Multi-Node – Classic

The most common multi-node deployment is:

- One node holds all modules as a Single Node deployment will do.
- All other nodes have the RDB and the SNA

<Frame>
  ![Image](/images/image-2.png)
</Frame>

In this case,  RegattaDB gets the resources available and utilizes all the nodes.

#### Multi-node - Standalone

In this case, we dedicate a node to the SM and also distribute the non-RDB Operational Modules across different nodes.

<Frame>
  ![Image](/images/image-5.png)
</Frame>

Note that since we place the SM module alone on this node, we don’t need an SNA. This topology may be common in some use cases, such as Kubernetes (K8) deployments.

For guidelines on sizing and configuring the RegattaDB to cater to your needs, please refer to \<\<\<BEST PRACTICES\>\>\> or contact support for assistance.

### Hardware Requirements

You will now need node to accommodate you RegattaDB configuration. These nodes may be physical, VMs or even containers. They need to at least meet all the prerequisites defined previously:

- CPUs
- RAM
- Storage devices Networking

To understand the exact requirements, you will need to size the system based on your workload needs. You can consult our \<\<\<BEST PRACTICES\>\>\> documentation or contact Regatta support.

### Regatta Software

Next, download the RegattaDB software package. RegattaDB is distributed as a set of RPM packages, each corresponding to a specific module or tool. These RPMs should be copied to the nodes where they will be installed in later steps based on defined configuration.

Below is an example of the content of the package file.

#### Deployment Software

Each of the RPMs below corresponds with one of the 6 RegattaDB modules

```shellscript
regatta-dcm-26.0.0.112-1.el8.x86_64.rpm
regatta-gdd-26.0.0.112-1.el8.x86_64.rpm
regatta-rdb-26.0.0.112-1.el8.x86_64.rpm
regatta-sequencer-26.0.0.112-1.el8.x86_64.rpm
regatta-sm-26.0.0.112-1.el8.x86_64.rpm
regatta-sna-26.0.0.112-1.el8.x86_64.rpm
```

#### Tools

The Regatta Support tools:

- Backup & Restore tool (BnR)
- Regatta Deployment Tool (RDT)

```shellscript
regatta-bnr-26.0.0.112-1.el8.x86_64.rpm
regatta-rdt-26.0.0.112-1.el8.x86_64.rpm
```

#### Clients

Regatta connectivity tools:

- Regatta Client library - required for any connectivity software.
- Regatta Command Line Interface (CLI).
- Regatta Connect – A full suite of drivers and connectors for integrating RegattaDB with external systems and frameworks (drivers, ORMs, etc.).

```shellscript
regatta-cli-26.0.0.112-1.el8.x86_64.rpm
regatta-client-26.0.0.112-1.el8.x86_64.rpm
regatta-connect-26.0.0.112-1.el8.x86_64.rpm
```

#### Debugging Tools

```shellscript
regatta-debug-26.0.0.112-1.el8.x86_64.rpm
```

### EULA Acceptance

To deploy and use RegattaDB, you must read and accept the **Regatta End User License Agreement (EULA)**. This acceptance is built directly into the RPM installation process; you will be prompted to agree to the terms before the installation can be completed.

- **Interactive Installs** (e.g., yum install, dnf install): you will be prompted to accept the EULA during installation.
- **Automated Installs**: You must set the environment variable ACCEPT\_EULA=1 to proceed without prompts.

Please take a few minutes to read and approve the Regatta End User License Agreement at the Regatta website ([https://regatta.dev/eula/](https://regatta.dev/eula/)) before deployment, especially if you are using the environment variable to bypass interactive prompts.

### Deployment User

In the installation process, the RPM creates a user named **regatta** with required permissions and establishes the directory structure. If you prefer to use another user, say **my-user**, please perform the following steps:

- Make sure **my-user** has the required permissions
- Transfer ownership of the created directory structure
- Add **my-user** to the required permissions groups (if missing)
- Update the System Configuration files for both SM and SNA

### IP Assignment

Identify the nodes and prepare a list containing Internal IPs for cluster communication and External IPs for application access.

We also need to assign 2 ports to each of the modules, one for communications and the other for support related operations.  While the Regatta system is flexible and allows the use of any available port, we recommend using the following standardized numbers whenever possible to ensure consistency across the deployment.

| Module | Communications | Support |
| :-- | :-- | :-- |
| SM | 8840 | 5000 |
| SNA | 8841 | 5001 |
| Sequencer | 8842 | 5002 |
| GDD | 8843 | 5003 |
| DCM | 8844 | 5004 |
| RDB | 8850 | 6001 |

### Firewall Setup

Configure the firewall on all nodes to allow access to internal ports and external application access.
